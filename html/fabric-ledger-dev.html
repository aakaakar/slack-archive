@mart0nix Query() API in chaincode going away has no material impact. Let me explain. In v0.5 Invoke() automatically went to all peers for execution and Query() went to one peer for query. So we can still do queries within a chaincode Invoke(), and we will actually likely open up query to peer API (outside chaincode), if you are a user authorized to see all blockchain data (for example an auditor). @binhn @garisingh We have talked about three things in this space to control query access: 1) Ability to disable GetQueryResult() peer (qscc) API for cases where all queries should go through application chaincode 2) Within application chaincode, the query will be scoped to that chaincode only. This will be enforced using the envelope data wrapper that was recently introduced over the couch key value data. The envelope will include ‘chaincodeid’ and in the query we will inject a constraint where chaincodeid=<current chaincode context>. We could also decide to use a couch database per chain-chaincode, rather than a couch database per chain (like we do now). 3) Application chaincode can use parameterized query as in the marbles02 example, rather than allowing query string to be passed in. Parameters are passed into the chaincode. The query is defined in the application chaincode and only the allowed parameters are substituted in (similar to jdbc query parameters used in java application code).
