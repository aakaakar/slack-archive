weeds
2016-11-15 19:09
1. Genesis tx of channels should contain at least one entry point / anchor address - Anchor peer: the peer responsible for getting updates from the ordering service on a certain channel transactions and routing them to that channel's gossip network; anchor peer is either defined by a domain name, so that it is easily substitutable. Domain names are included in the certs. - Approach 1: a) anchor peers defined by the configuration application and included in the genesis block of a channel b) Configuration application sends join channel request to all the peers they want to obtain updates on that channel c) Peers with a JoinChannel request contact the ordering service to get the genesis block d) Because genesis block contains anchor peers, the peer knows which anchor peer to contact for that channel. - Approach 2: anchor peers are defined with the rest of participant peers that participate in a channel at genesis time.

weeds
2016-11-15 19:09
- Approach 3: a) anchor peers defined by the configuration application and included in the genesis block of a channel b) Configuration application sends join channel request to all the peers they want to obtain updates on that channel with the anchor peers c) Peers with a JoinChannel request contact the anchor peer to get updates on that chainnel d) Because genesis block contains anchor peers, the peer verifies the anchor id in the genesis block, while the anchor peer knows what are the peers it will gossip channel updates. e) Trust is put to the few anchor peers. Note: authentication of server side of anchor peers; how is this done? Overall leaning to approach 3. 2. Bootstrap - Orderer: MSPs, orderers, Policies => config block - Peer: ID, {IP: port, msp keystore} => yaml; the rest can be added after the peer has rebooted. Currently an endorser processes all proposals it receives even if they refer to chaincodes they are not in their endorsement policy in. Endorsement policies should be possible to be based on participant identities. 3. Genesis tx of channels participant information Each channel has its own MSPs and their names have as namespace the name of the channel they belong to. Genesis of each channel would need to contain the full definition of an MSP. 4. Q: Do we have the orderers process all configuration tx-s and filter out invalid ones before sending this to each channel? There was a remark from Jason that orderers filter out invalid config transacitons before handing these to the endorser network. This is to be further discuss to my understanding. 5. Endorser epoch re-definition: - epoch numbers are defined in a sliding window manner - epoch starting point (in terms of block number) is added by the (submitting) peer to what is endorsed - epoch size is a config parameter - clients send to rest of peers a proposal that includes the epoch starting point as received by submitting peer - endorsers need to check if the starting point of epoch (included in the proposal) + epoch window < current block number and then respond - change on the protocol required from the client to peer to include an optional epoch number in the proposal ( same spirit as with the simulation results) 6. Revocation: OCSP based verification ; could have asynchrony problems but maybe turn out to be a solution in certain cases. - we start with CRLs advertised in the Blockchain and we see where we go from there 7. DeliverRequest authentication Replay attack for deliver requests is important, as anyone who can replay the deliver request to a peer would essentially be able to bypass the ACLs associated to that channel. To avoid this TLS (that already implements replay attack protection) should be enabled, and at the application layer we use TLS binding in the deliver request signature. mandler

The way we thought about it in the context of gossip: 1. Gossip selects internally a leader peer for every organization (Member) - equivalent to an anchor peer 2. The leader connects to the ordering service on behalf of the organization 3. JoinChannel is invoked on a peer - the information is propagated within the organization peers, and in particular reaches the organization leader which invokes Deliver() on the ordering service 4. Blocks flow from the ordering service through the leader and are gossiped to all registered peers in the organization 5. Information on peer inclusion in a channel is gossiped with all other peers in the designated organizations 6. State transfer / synchronization within a channel can take place across organizations belonging to the same channel 7. Upon a leader failure a new one is selected within the organization and connects to the ordering service on behalf of all registered channels More detailed information can be found in the gossip design document: https://docs.google.com/document/d/157AvKxVRqgeaCTSpN86ICa5x-XihZ67bOrNMc5xLvEU/edit#heading=h.rrii36vrca54
